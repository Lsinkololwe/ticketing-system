CRM UAT Kubernetes Network Architecture

Environment: Three-node on-premise UAT cluster

Date: 10 November 2025

Prepared for: Network Infrastructure Team

Prepared by: Digital & Enterprise Architecture Team, Zambia National Commercial Bank

Executive Overview

The CRM UAT Kubernetes cluster will run on three nodes located on the corporate VLAN 10.253.11.0/24. All address allocations conform to RFC 1918 guidance. HAProxy terminates inbound traffic and forwards NodePort 30080 to the nodes, while pod traffic is masqueraded behind node IPs. No additional routes or firewall changes are required beyond the HAProxy configuration. Please reserve the following blocks to prevent overlap with other internal systems:

  • Pod CIDR: 10.200.0.0/22
  • Service CIDR: 192.168.100.0/24

If these are unavailable, any equivalent RFC 1918 /22 (pods) and /24 (services) can be substituted without affecting the design.

Node Inventory

  • Control Plane Node: 10.253.11.230 – runs the API server, scheduler, and controller manager; application pods are tainted off this node.
  • Worker Node A: 10.253.11.235 – exposes NodePort 30080 and schedules CRM pods such as App1 (10.200.1.10) and App2 (10.200.1.11).
  • Worker Node B: 10.253.11.124 – also exposes NodePort 30080 and hosts App3 (10.200.2.20) and App4 (10.200.2.21).

The shared ClusterIP service crm-service (192.168.100.50) fronts every CRM pod across the worker nodes while HAProxy continues to target the node IPs via NodePort 30080.

Traffic Flow Examples

Inbound (Client → Pod)

Client → HAProxy 10.253.11.240:30080 → Worker Node A NodePort 30080 (10.253.11.235) → crm-service 192.168.100.50 → Pod 10.200.1.10.

Outbound (Pod → External API)

Pod 10.200.2.20 → SNAT → Node IP 10.253.11.124 → Corporate network/external service. Upstream systems see the node IP as the source.

East-West (Service Discovery)

Worker pod 10.200.1.11 resolves crm-service.default.svc.cluster.local → ClusterIP 192.168.100.50 → kube-proxy load-balances to healthy CRM pod IPs on both worker nodes.

Security & Firewall Considerations

  • No static routes or BGP advertisements are necessary; overlay handles pod reachability.
  • Permit inter-node communication for the pod overlay (CNI uses encapsulation or routing on ports specific to the chosen plugin, commonly IP-in-IP or VXLAN).
  • Allow NodePort range 30000-32767 plus any ingress ports (typically 80/443) on node IPs.
  • Optional: Kubernetes NetworkPolicies can restrict pod-to-pod traffic to enforce least privilege.

With these ranges reserved, platform engineering can proceed with cluster bootstrap and handover detailed runbooks to the UAT operations team.
